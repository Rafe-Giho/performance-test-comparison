# 부하테스트 도구 비교 보고 가이드

## 문서 등급

문서 등급: 팀 배포용 표준 운영 가이드

## 운영 환경

운영 환경: 클라우드 기반 테스트 환경

이 문서의 구축, 실행, 검증 기준은 클라우드 기반 테스트 환경을 전제로 한다. 테스트 인프라는 프로젝트별 클라우드 VM, 컨테이너, Kubernetes/NKS 자원으로 구성하고 운영 서비스와 논리적으로 분리한다.

이 문서는 비교 보고서의 결론, 평가표, 최신 상태 근거를 팀 배포 가능한 기준으로 정리한다.

## 목적

이 문서는 `JMeter`, `nGrinder`, `k6`를 비교해 상신 보고서를 작성할 때 사용할 기준서다.

이 문서의 목표는 아래 4가지를 한 번에 해결하는 것이다.

- 세 도구를 같은 조건으로 비교하는 방법 제시
- PoC를 어떻게 설계하고 증빙할지 제시
- 보고서에 어떤 항목을 넣어야 하는지 정리
- 우리 조직 기준의 결론과 권고 방향을 제시

## 보고서에서 먼저 답해야 할 질문

보고서는 단순 기능 나열이 아니라 아래 질문에 답해야 한다.

1. 우리 조직의 기본 표준 도구는 무엇인가
2. 어떤 고객과 어떤 프로젝트에 어떤 도구가 더 적합한가
3. 왜 그 결론이 나왔는가
4. 지금 보유 자산은 어떻게 활용하고 무엇은 줄여야 하는가

## 보고서의 기본 결론 방향

현재 조직 조건을 반영하면 보고서의 기본 결론은 아래 방향이 가장 일관된다.

- 공통 표준 도구: `JMeter`
- 전략 특화 도구: `k6`
- 한시적 운영 자산: `nGrinder`

이 결론의 이유는 간단하다.

- `JMeter`: 공공기관 설득력, 범용성, 익숙함, 수행자 교체 대응력
- `k6`: API/클라우드 네이티브/자동화/CI 친화성
- `nGrinder`: 기존 자산 활용 가치는 있으나 장기 표준화 리스크 존재

## 현재 자산 검증 상태

세 도구는 같은 수준으로 검증된 것이 아니므로 보고서에서는 아래 상태를 구분해서 적는다.

| 도구 | 완료된 검증 | 보고서에서의 표현 기준 |
| --- | --- | --- |
| JMeter | 클라우드 VM/분산 운영 기준 문서, 표준 `.jmx`, 실행 스크립트, Docker runner 자산, 하네스 검증 | 조직 공통 표준 후보의 문서/자산 정합성 확보 |
| k6 | NKS에서 k6 Operator `4.3.2` 설치, CRD 확인, no-op `TestRun`, server dry-run, 적용 스크립트 dry-run | 클라우드 네이티브 실행 체인 검증 완료 |
| nGrinder | Docker Desktop에서 Controller/Agent Compose 기동, Controller HTTP `200`, Agent 연결 로그 확인 | Compose 기반 단기 운영 체인 검증 완료 |

대상 시스템에 실제 부하를 발생시키는 smoke/load/stress/spike/soak는 세 도구 모두 프로젝트별 승인된 URL, 계정, payload, 모니터링 연결 후 수행한다.

## 비교 방법과 방안

세 도구 비교는 `기능 비교`, `실행 비교`, `도입 비교`를 분리해서 해야 한다.

### 1. 기능 비교

도구가 무엇을 시험할 수 있는지 본다.

- 웹 페이지 조회
- 로그인/세션 유지
- 목록/상세 조회
- 핵심 이벤트성 API 호출
- spike, stress, soak 같은 부하 패턴
- 분산 실행 가능 여부
- 결과 리포트 생성 방식

### 2. 실행 비교

같은 시나리오를 실제로 구현하고 돌려 본다.

- 스크립트 또는 테스트 플랜 작성 시간
- 동적 값 처리 난이도
- smoke test 성공 여부
- load/stress/spike 구현 난이도
- 분산 실행 구성 시간
- 결과 리포트 해석 용이성

### 3. 도입 비교

조직과 고객 입장에서 실제 도입 적합성을 본다.

- 공공기관 보고 설득력
- 익숙한 레퍼런스 여부
- 인수인계 난이도
- 클라우드 네이티브 적합성
- 운영 표준화 가능성
- 장기 유지보수 리스크

## 공정 비교 원칙

비교가 설득력을 가지려면 아래 조건을 반드시 맞춰야 한다.

- 동일한 테스트 대상
- 동일한 핵심 시나리오
- 동일한 부하 단계
- 동일한 성공 기준
- 동일한 메트릭 수집 범위
- 동일한 인프라 조건

아래처럼 비교하면 안 된다.

- 도구마다 다른 대상 시스템을 씀
- 어떤 도구는 API만, 어떤 도구는 브라우저까지 포함함
- 어떤 도구는 단일 노드, 어떤 도구는 분산으로 비교함
- 성공 기준이 다름

## 추천 PoC 설계

가장 비교가 잘 되는 기준 시나리오는 `공공기관 홈페이지 + 로그인/API 혼합`이다.

### 공통 시나리오

1. 메인 또는 health 진입
2. 로그인
3. 목록 조회
4. 상세 조회
5. 핵심 이벤트 API 호출
6. 종료

### 공통 부하 단계

1. `Smoke`
2. `Load`
3. `Stress`
4. `Spike`
5. `Soak`

### 공통 측정 메트릭

- 평균 응답시간
- P90/P95/P99
- 에러율
- TPS/RPS
- CPU
- 메모리
- 네트워크
- DB connection
- slow query
- LB/Ingress/WAF 병목 여부

## 비교 결과 항목

보고서에는 아래 항목이 반드시 들어가야 한다.

### A. 구축 관점

- 설치 방식
- 기본 토폴로지
- 최소 시작 사양
- 분산 실행 구조
- 클라우드 인프라 준비 난이도
- 이미지/버전 고정 용이성

### B. 테스트 수행 관점

- 스크립트 작성 난이도
- GUI 또는 코드 생산성
- 동적 파라미터 처리
- 인증/세션 처리
- 시나리오 재사용성
- spike/stress/soak 표현력
- 결과 리포트 품질

### C. 운영 관점

- 운영 표준화 용이성
- 수행자 교체 대응력
- Git 기반 협업성
- CI/CD 연계성
- 산출물 재사용성
- 템플릿화 용이성

### D. 고객 설득 관점

- 공공기관 담당자 신뢰도
- 오래된 레퍼런스와 익숙함
- 보고서 설명 용이성
- 산출물 가시성
- GUI/웹 UI 기반 설명 가능성

### E. 전략 관점

- WEB-WAS-DB형 IaaS 적합성
- Kubernetes/MSA 적합성
- API 중심 사업 적합성
- 장기 유지보수성
- 생태계/커뮤니티 지속성
- 보안/업데이트 리스크

## 보고서 작성 방향

보고서는 `결론 -> 근거 -> 적용 방안` 순으로 써야 한다.

### 1. 결론부터 쓴다

첫 페이지에서 아래를 바로 말해야 한다.

- 우리 조직의 공통 표준은 무엇인지
- 전략 특화 도구는 무엇인지
- 기존 도구는 어떤 조건에서 유지할지

### 2. 비교 기준을 먼저 공개한다

도구별 장단점을 나열하기 전에 아래를 먼저 써야 한다.

- 동일 대상
- 동일 시나리오
- 동일 부하 단계
- 동일 메트릭
- 동일 평가표

이걸 먼저 적어야 `주관적 선호`가 아니라 `기준 기반 결론`처럼 보인다.

### 3. 고객 유형별 권고를 분리한다

하나의 도구를 모든 고객에게 밀면 설득력이 떨어진다.

반드시 아래처럼 나눠 써야 한다.

- 공공기관 기본 홈페이지/WEB-WAS-DB형 IaaS: `JMeter`
- Kubernetes/MSA/API 중심 사업: `k6`
- 기존 내부 자산 즉시 활용 프로젝트: `nGrinder`

### 4. 최종 방향은 조직 표준 관점으로 닫는다

보고서 마지막에는 아래 방향으로 닫는 것이 가장 자연스럽다.

- 조직 공통 표준: `JMeter`
- 전략 특화 및 자동화 축: `k6`
- 기존 자산 활용 및 단기 대응: `nGrinder`

## 보고서에 넣을 현재 운영 현황 예시

- 기존에는 `JMeter`로 부하테스트를 수행해 왔다.
- 최근에는 `nGrinder`를 통해 부하테스트를 수행했다.
- 이번 표준 자산 정비에서 JMeter는 문서/자산 검증, k6는 NKS Operator 실행 체인 검증, nGrinder는 Docker Compose 기동 검증까지 수행했다.
- 공공기관 고객 비중이 높아 익숙하고 설명 가능한 도구가 중요하다.
- 최근에는 `Kubernetes` 기반 클라우드 네이티브 전환 사업이 증가하고 있다.
- 동시에 `WEB-WAS-DB` 구조의 IaaS형 고객도 여전히 많다.
- 따라서 단일 도구만의 우열보다 `공통 표준 + 전략 특화` 관점이 필요하다.

## 예상 비교 결과 정리

### JMeter

- 가장 강한 포지션: `공통 표준`
- 이유:
  - 공공기관 보고 설득력
  - 높은 범용성
  - 기존 조직 경험과의 연속성
  - GUI 기반 설명 가능성

### nGrinder

- 가장 강한 포지션: `기존 자산 활용`
- 이유:
  - Compose 기반 빠른 재현
  - Controller-Agent 구조
  - 중앙 관리 편의성
- 단, 신규 장기 표준 후보로는 리스크가 크다.

### k6

- 가장 강한 포지션: `전략 특화`
- 이유:
  - API 중심 테스트
  - 클라우드 네이티브 적합성
  - 코드 기반 협업
  - 자동화/CI 친화성

## 최종 권고 문안 예시

> 비교 결과, 우리 조직의 공통 표준 부하테스트 도구는 JMeter로 유지하는 것이 가장 적절하다. JMeter는 공공기관 고객에게 설명 가능성이 높고, WEB-WAS-DB형 IaaS 사업과 혼합 프로토콜 환경에 폭넓게 대응할 수 있다. 다만 Kubernetes/MSA/API 중심의 클라우드 네이티브 사업이 증가하는 점을 고려해, 전략 특화 도구로 k6를 병행 육성하는 것이 바람직하다. nGrinder는 현재 보유 자산을 활용하는 단기 운영 도구로는 유효하나, 장기 표준화 대상보다는 한시적 운영 자산으로 관리하는 것이 적절하다.

## 보고서 작성 시 주의사항

- 도구 소개서처럼 쓰지 말고 `우리 조직에 무엇이 맞는가` 중심으로 쓴다.
- 기능 나열보다 `도입 판단 근거`를 먼저 쓴다.
- 공공기관 고객 관점과 내부 기술조직 관점을 분리해서 설명한다.
- `한 도구가 모든 상황에 최고`라는 식으로 쓰지 않는다.
- 최종 권고는 반드시 `고객 유형별 적용 기준`과 함께 제시한다.
