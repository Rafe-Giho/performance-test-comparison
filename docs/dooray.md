# 부하테스트 도구 간단 비교 분석 표

## 문서 등급

문서 등급: 팀 배포용 표준 운영 가이드

## 운영 환경

운영 환경: 클라우드 기반 테스트 환경

이 문서의 구축, 실행, 검증 기준은 클라우드 기반 테스트 환경을 전제로 한다. 테스트 인프라는 프로젝트별 클라우드 VM, 컨테이너, Kubernetes/NKS 자원으로 구성하고 운영 서비스와 논리적으로 분리한다.

| **비교 항목** | **JMeter** | **nGrinder** | **k6** |
| ----- | ------ | -------- | --- |
| 기본 철학 | GUI 중심 + 필요 시 스크립트 확장 | 중앙 웹 UI 중심의 조직형 성능 테스트 플랫폼 | 코드 우선(Code-first) 성능 테스트 |
| 주요 작성 방식 | GUI Test Plan(`.jmx`) + JSR223(Groovy 등) | 웹 UI 또는 IDE 기반 Groovy/Jython 스크립트 | JavaScript/TypeScript 스크립트 |
| 시나리오 표현력 | 기본 흐름은 쉽지만 복잡 로직은 JSR223 의존도가 높음 | 코드 기반으로 표현 가능하나 언어 생산성과 병렬 제어는 다소 제한적 | 세션 유지, 랜덤 선택, 조건 분기, 재사용까지 가장 자연스럽게 표현 가능 |
| 멀티 API / 병렬 호출 | 가능하지만 구성 복잡도가 올라감 | 가능하지만 구현 유연성은 k6보다 낮음 | `http.batch()` 등으로 가장 유연하게 구성 가능 |
| 강한 영역 | 레거시 포함 다양한 프로토콜, 빠른 레코딩, 폭넓은 레퍼런스 | 여러 사용자가 공유하는 사내 플랫폼, 에이전트 분산 실행 | API 테스트, CI/CD, GitOps, 자동화, 성능 기준선 관리 |
| 지원 프로토콜 성격 | HTTP/HTTPS, SOAP/REST, FTP, JDBC, LDAP, JMS, Mail, TCP 등 폭넓음 | HTTP 중심이지만 라이브러리 확장으로 타 프로토콜 대응 가능 | HTTP/2, WebSocket, gRPC, TLS, Browser 모듈 등 현대 웹/API 중심 |
| 브라우저 관점 | 브라우저가 아니라 프로토콜 레벨 테스트 도구 | 브라우저 자동화 자체가 핵심은 아님 | `k6/browser` 제공, 브라우저 수준 시나리오 가능 |
| 브라우저 테스트 성격 | 실제 렌더링/DOM/JS 실행 측정은 불가 | 실제 브라우저 UX 측정보다는 HTTP 부하 중심 | `k6`는 대규모 HTTP 부하, `k6/browser`는 UX/렌더링/기능 검증에 적합 |
| 스크립트 버전관리 | 가능하지만 GUI 산출물 diff/merge는 불리한 편 | 스크립트 저장소와 중앙관리 구조에 강점 | 텍스트 기반 코드라 Git diff/PR 협업에 가장 유리 |
| 분산 실행 | Remote/Distributed Testing 가능, 운영 편의성은 낮은 편 | Controller-Agent 구조가 기본, 다중 에이전트/다중 지역 실행 강점 | 단일 VM 또는 k8s operator 가능, OSS는 별도 분산 운영 구성이 필요하고 Grafana Cloud k6는 관리형 인프라 제공 |
| 결과 분석 | HTML Dashboard, 다수 플러그인, 로컬 분석 친화적 | 웹 UI 기반 모니터링과 결과 관리 강점 | Threshold, 실시간 출력, 다양한 출력 연동, Grafana 연계 강점 |
| CI/CD 적합성 | 가능하지만 GUI 중심 자산이 누적되면 불리 | 가능하나 조직형 운영 도구 성격이 강함 | 가장 적합. 코드형 테스트와 자동 판정 구조가 명확 |
| GUI / 코드 중심 | GUI 중심이지만 CLI 실행과 스크립트 확장 가능 | 중앙 웹 UI와 코드 작성이 함께 가는 조직형 플랫폼 | 본질적으로 코드형이며 GUI(k6 Studio)는 보조 수단에 가까움 |
| 러닝 커브 | 초반 진입 쉬움, 복잡해질수록 Test Plan 관리 난이도 증가 | 설치와 운영 개념을 이해해야 하며 Controller-Agent 구조 학습 필요 | 개발자에게 친숙, 비개발자에게는 코드 작성 장벽이 있을 수 있음 |
| 확장성 | 플러그인과 JSR223 확장이 매우 풍부 | 라이브러리 추가와 커스텀 의존성 사용이 유연 | 확장(extension) 생태계와 출력 연동이 좋음 |
| 리소스 요구량 | JVM 기반이라 설정에 따라 무거워질 수 있음 | Controller + Agent 플랫폼 운영 관점에서 오버헤드가 큼 | HTTP 기준 가장 가벼운 편이나 `k6/browser`는 가장 무거운 축에 속함 |
| 장기 유지보수 관점 | 안정적이고 보수적 | 신규 채택 리스크 큼 (유지보수 마지막 기록 25년도) | 현재 가장 현대적이고 운영 자동화에 유리 |

## 현재 자산 검증 상태

| 도구 | 완료된 검증 | 남은 프로젝트 검증 |
| ----- | ------ | -------- |
| JMeter | 클라우드 VM/분산 운영 기준 문서, 표준 `.jmx`, 실행 스크립트, Docker runner 자산, 하네스 검증 | 실제 VM 설치, 단일/분산 smoke, 실제 대상 부하 실행 |
| nGrinder | Docker Desktop Compose 기동, Controller HTTP `200`, Agent 연결 로그 | 클라우드 VM 배포, Controller UI smoke, 실제 대상 부하 실행 |
| k6 | NKS k6 Operator `4.3.2` 설치, CRD 확인, no-op `TestRun`, server dry-run, 적용 스크립트 dry-run | 실제 `BASE_URL`, Secret, payload 기반 API/browser smoke와 본시험 |

검증에 사용한 런타임 자원은 정리했으며, 현재 저장소에는 문서와 실행 자산만 남긴다.

## 주요 특징(장단점) 정리

### **1\. JMeter**

* 장점

    * 공식 페이지 기준 지원 프로토콜 범위가 세 도구 중 가장 넓다.
    * GUI Test IDE, 레코딩, 디버깅, HTML 리포트가 잘 갖춰져 있어 PoC 시작이 빠르다.
    * HTTP 외에도 JDBC, JMS, FTP, Mail, TCP 등 이질적인 시스템을 한 도구로 묶기 좋다.
    * GUI 기반으로 기본 시나리오를 빠르게 조립할 수 있고, CLI 실행도 가능하다.
* 단점

    * 테스트 자산이 커질수록 GUI 중심 산출물 관리가 어려워진다.
    * 복잡한 사용자 시나리오, 랜덤 선택, 상태 의존 흐름은 JSR223(Groovy) 비중이 커져 유지보수가 어려워진다.
    * 분산 실행은 가능하지만 여러 노드 버전 정합성, 네트워크 설정, 원격 서버 관리 부담이 존재한다.
    * 공식 문서가 명시하듯 브라우저 자체를 실행하지 않으므로 실제 프론트엔드 렌더링 성능 측정 도구로 보면 한계가 있다.
    * JVM 기반이라 리스너나 GUI 설정에 따라 리소스 사용량이 빠르게 커질 수 있다.
* 적합한 조직

    * QA 중심
    * 레거시/혼합 프로토콜 환경
    * 빠른 시각적 설계와 레코딩이 필요한 팀

### **2\. nGrinder**

* 장점

    * Controller-Agent 구조와 웹 UI가 기본이라 사내 공용 성능 테스트 플랫폼처럼 운영하기 좋다.
    * 공식 소개 기준 다중 에이전트, 다중 지역 배치, 동시 테스트 실행, 결과 관리에 강점이 있다.
    * Jython 또는 Groovy 기반이며 외부 라이브러리와 Maven 의존성 확장이 가능하다.
    * 세션 유지, 단계 의존 흐름, 랜덤 선택 같은 시나리오를 코드로 표현할 수 있어 JMeter보다 흐름 제어가 자연스럽다.
* 단점

    * 가장 큰 이슈는 유지보수 신호다. 공식 GitHub 저장소가 `2025-09-24`에 archive 되었고 read-only 상태다.
    * GitHub 최신 릴리스는 `3.5.9-p1`이지만 archive 상태이므로, 신규 도입 시 운영/보안/호환성 검토를 더 엄격히 해야 한다.
    * 현대적인 DevOps 파이프라인 표준 도구로 밀기에는 생태계 활력이 약하다.
    * 여러 API 호출과 병렬 흐름 구성은 가능하지만, 코드 표현력과 구현 유연성은 k6보다 낮다.
    * 실제 브라우저 렌더링/UX 측정보다는 HTTP 부하 중심이며, Controller + Agent 플랫폼 운영 오버헤드가 있다.
* 적합한 조직

    * 이미 nGrinder 경험이 있는 팀
    * 사내 온프레미스 다중 에이전트 플랫폼이 꼭 필요한 조직
    * 운영 플랫폼 관점이 중요하고 신규 표준화보다 기존 자산 활용이 우선인 경우

### **3\. k6**

* 장점

    * JavaScript/TypeScript 기반의 코드형 테스트라 Git, PR, 코드리뷰, CI/CD에 가장 잘 맞는다.
    * 공식 문서 체계상 `Thresholds`, `Checks`, `Scenarios`, `Results output`가 선명하게 정리되어 있어 자동 판정 파이프라인 구축이 쉽다.
    * `HTTP/2`, `WebSockets`, `gRPC`, `k6/browser` 등 현대 웹/API 테스트 포인트에 잘 맞는다.
    * Grafana Cloud k6를 쓰면 분산 인프라 스케일링과 메트릭 저장/집계를 관리형으로 처리할 수 있다.
    * 단일 VM 방식과 k8s operator 방식으로 실행 가능하며 대규모 테스트에도 유용하다.
    * 세션 유지, JSON 파싱, 랜덤 선택, 조건 분기, 멀티 API 호출, 병렬 시나리오 구성이 가장 자연스럽다.
    * HTTP 기준 세 도구 중 가장 가벼운 편이라 대규모 API 성능 테스트에 유리하다.
* 단점

    * GUI로 빠르게 찍어내는 방식보다 코드 작성 전제가 강하다.
    * OSS만으로 대규모 분산 실행을 하려면 Kubernetes Operator 등 별도 운영 구성이 필요하다.
    * DB/JMS/레거시 엔터프라이즈 프로토콜 범용성은 JMeter보다 약하다.
    * `k6/browser`는 실제 브라우저 기반이라 UX/렌더링/기능 검증에는 좋지만 대규모 부하 발생기로 쓰기에는 매우 무겁다.
    * Operator 환경에서 browser 테스트를 하려면 로컬처럼 단순 실행이 아니라 브라우저 포함 이미지와 별도 실행 구성이 필요하다.
* 적합한 조직

    * 개발팀 주도 성능 테스트
    * API 성능 테스트 자동화
    * CI 파이프라인에서 SLA/Threshold 미달 시 즉시 실패 처리하려는 팀

## **추가 정리: 이번 비교에서 확인한 핵심 차이**

### **1\. 같은 사용자 시나리오를 구현할 때의 방식 차이**

로그인 → 마이클래스 → 강의실 홈 → 학습하기 → 강의영상 재생 같은 흐름은 세 도구 모두 구현 가능하다. 다만 구현 방식은 다르다.

* **JMeter**

    * Thread Group 아래에 HTTP Request, Cookie Manager, JSON Extractor, Timer, JSR223 등을 트리 형태로 배치해 구성한다.
    * 화면상 조립은 쉽지만, 과목/회차 랜덤 선택 같은 로직이 들어가면 JSR223(Groovy) 의존도가 높아진다.
* **nGrinder**

    * Groovy/Jython 스크립트에서 로그인, 세션 유지, 랜덤 선택, sleep 기반 think time을 코드로 구현한다.
    * JMeter보다 코드 흐름이 자연스럽지만 언어 생산성과 생태계는 약한 편이다.
* **k6**

    * JavaScript 코드로 사용자 시나리오를 그대로 표현할 수 있어 가장 자연스럽다.
    * 세션 유지, JSON 파싱, 랜덤 선택, 조건 분기, 시나리오 재사용이 가장 쉽다.

### **2\. 여러 API를 동시에 호출하는 능력 차이**

* **JMeter / nGrinder / k6 모두 여러 API 호출 시나리오 구성은 가능하다.**
* 다만 **k6가 가장 유연하다.**

    * `http.batch()` 같은 방식으로 병렬 호출을 쉽게 표현할 수 있다.
    * JavaScript 기반이라 조건 분기와 복합 시나리오 구현이 쉽다.
* **nGrinder도 불가능한 것은 아니지만**, 코드 표현력과 병렬 흐름 제어 측면에서는 k6보다 불편하다.
* 따라서 정확한 표현은 \*\*"nGrinder는 못 한다"가 아니라, "k6가 훨씬 쉽게 구현한다"\*\*가 맞다.

### **3\. 브라우저 테스트 관점 차이**

* **JMeter / nGrinder**

    * 기본적으로 HTTP/프로토콜 레벨 테스트 도구다.
    * 실제 브라우저 렌더링, DOM 처리, JavaScript 실행, 사용자 클릭 기반 흐름은 측정하지 않는다.
* **k6/browser**

    * Chromium 기반 실제 브라우저 흐름을 사용한다.
    * 렌더링, JS 실행, 클릭, 입력, 페이지 이동 같은 사용자 체감 흐름을 검증할 수 있다.
    * 따라서 서버 부하 자체보다 **프론트엔드 렌더링 성능, UX 체감, 기능 동작 확인**에 더 가깝다.

### **4\. k6 와 k6/browser의 역할 분리**

* **k6 (기본 HTTP)**

    * 대규모 부하 테스트용
    * 수천\~수만 VU 테스트에 적합
    * 서버/API 성능, TPS/RPS, 병목 구간 파악에 적합
* **k6/browser**

    * 브라우저 기반 사용자 시나리오 검증용
    * 실제 사용자가 느끼는 화면 전환, 로딩, JS 렌더링 검증에 적합
    * 리소스 요구량이 매우 커서 대규모 부하 발생기로 쓰기엔 비효율적
* 정리하면 **k6/browser는 순수 부하테스트 도구라기보다 UX/렌더링/기능 동작 검증 도구**로 보는 것이 더 정확하다.

### **5\. k6 Operator와 browser 테스트의 관계**

* 로컬에서는 `k6 new --template browser ...`로 브라우저용 스크립트를 만든 뒤 `k6 run`으로 바로 실행할 수 있다.
* 하지만 **k6 Operator는 CLI를 대신 치는 구조가 아니라 Kubernetes CRD 기반 실행 구조**다.

    * 스크립트를 ConfigMap/PVC/localFile 형태로 클러스터에 올리고
    * `TestRun` 리소스로 실행하며
    * runner 이미지는 브라우저 포함 이미지가 필요하다.
* 즉, **스크립트는 재사용 가능하지만 실행 방식은 로컬과 다르다.**
* Operator에서 browser 테스트를 하려면 일반 k6 이미지가 아니라 브라우저 실행 환경이 포함된 runner 이미지가 필요하다.

### **6\. 리소스 요구량 비교**

같은 HTTP 기반 시나리오만 비교하면 일반적으로 다음 경향이 있다.

* **가벼움:** k6 → JMeter ≈ nGrinder → **무거움**
* **k6**는 가장 가벼운 편이다.
* **JMeter**는 JVM 기반이라 GUI 요소나 리스너 설정에 따라 빨리 무거워질 수 있다.
* **nGrinder**는 Controller + Agent 플랫폼 전체 운영 관점에서 오버헤드가 더 크다.
* 하지만 **browser 테스트까지 포함하면 `k6/browser`가 세 가지 중 가장 무겁다.**

    * 실제 Chromium 실행
    * 높은 CPU/Memory 사용
    * Pod/프로세스당 소수 VU만 감당 가능

### **7\. GUI vs 코드 중심 차이**

* **JMeter**

    * GUI 기반 설계가 중심이지만 CLI 실행도 가능하다.
    * 즉, UI 전용 도구는 아니지만 기본 철학은 GUI 조립형에 가깝다.
* **nGrinder**

    * 중앙 웹 UI와 Agent 기반 운영 플랫폼 성격이 강하다.
    * 코드 작성도 가능하지만 조직형 플랫폼 관점이 더 강하다.
* **k6**

    * 본질적으로 코드형 도구다.
    * GUI 메인 도구라기보다 tests-as-code 철학에 가깝다.
    * 시각적 보조 도구는 있을 수 있어도, 기본은 JavaScript 스크립트 실행이다.

### **8\. 실무적으로 의미 있는 병행 방식**

* **nGrinder 또는 JMeter**로 대규모 HTTP 트래픽을 발생시켜 서버 한계와 병목을 본다.
* 동시에 **k6/browser**로 실제 사용자 화면 전환, 렌더링, 클릭 흐름, JS 지연을 측정한다.
* 이 조합은 매우 의미가 있다.

    * 서버는 버티는데 사용자는 느리게 느끼는 문제
    * API 평균 응답은 괜찮지만 프론트엔드 체감 성능이 나쁜 문제
    * 대규모 부하 상태에서 실제 화면 UX가 어떻게 악화되는 지를 같이 확인할 수 있기 때문이다.

## 수행 환경 및 대상 정의

**업무상 수행 환경**

* 우리 조직은 고객, 서비스, 수행자가 프로젝트마다 달라지는 MSP형 구조
* 테스트 인프라는 상시 고정 운영이 아니라 프로젝트별로 신규 구성하는 경우가 많음(KTC, NHNC에 구성)
* 레거시 WEB-WAS-DB형 IaaS 사업과 Kubernetes/MSA형 클라우드 네이티브 사업이 공존함
* 기존에는 JMeter로 부하테스트를 수행해 왔음
* 최근에는 nGrinder를 활용한 수행 사례가 있으며, Docker-Compose.yaml로 1 Controller - 10 Agent 구성도 보유하고 있음
* 향후 공공기관 대상 Kubernetes 기반 클라우드 네이티브 전환 사업과 API 중심 사업의 비중이 증가하는 추세임

**대상(타깃)**

* 주요 테스트 대상은 웹사이트 및 웹 API
* 주요 검증 목적은 이벤트성 폭주, 사용자 몰림, 백엔드 병목 구간 확인
* 고객은 공공기관 비중이 높음
* 공공기관 담당자는 오래된 레퍼런스, 운영 사례, 익숙한 프로그램에 대한 심리적 신뢰가 높은 편임

**도구 검토 시 고려사항**

* 기술적 우수성만이 아니라 공공기관 대상 설명 가능성과 설득력 필요
* 인수인계 용이성, 수행자 교체 대응력, 표준화 가능성 필요
* 레거시/IaaS 대응력과 클라우드 네이티브/API 대응력을 함께 고려해야 함
* 현재 보유한 경험과 자산 활용 가능성도 함께 반영해야 함

## 표준화 전략

### 1\. 기본 방향

현재 정리한 내용을 기반으로 판단하면, 부하테스트 도구 표준화 방향은 단일 도구 일원화보다 [공통 표준 + 전략 특화 + 기존 자산 활용]의 3축 구조로 가져가는 것이 가장 현실적일 수 있다.

* 우리 조직은 MSP형 구조로 프로젝트마다 고객, 서비스, 수행자가 바뀌므로, 누구나 빠르게 인수인계하고 재사용할 수 있는 공통 표준이 필요하다.
* 고객의 상당수가 공공기관이며, 익숙한 도구와 운영 사례, 설명 가능한 산출물에 대한 신뢰가 크므로 대외 설득력이 높은 도구가 필요하다.
* 동시에 향후에는 Kubernetes/MSA/API 중심의 클라우드 네이티브 사업 비중이 증가하므로, 현대적인 자동화 및 코드형 테스트 역량도 함께 확보해야 한다.
* 기존 수행 경험과 보유 자산을 완전히 버리기보다, 활용 가능한 자산은 단기적으로 유지하고 장기 표준화는 별도 축으로 정리하는 것이 효율적이다.

### 2\. 도구별 표준 포지션

#### JMeter

* 포지션: `조직 공통 표준 도구`
* 선정 이유

    * 공공기관 대상 설명 가능성과 심리적 신뢰도가 높음
    * WEB-WAS-DB형 IaaS, 레거시, 혼합 프로토콜 환경에 대응 범위가 넓음
    * GUI 기반 작성과 HTML 리포트 등으로 수행자 교체 및 인수인계가 상대적으로 용이함
    * 기존 조직 경험과 연속성이 있어 표준화 초기 정착 비용이 낮음

#### k6

* 포지션: `전략 특화 도구`
* 선정 이유

    * API 중심, 클라우드 네이티브, Kubernetes/MSA 환경에 가장 잘 맞음
    * 코드 기반 테스트, Git 협업, CI/CD 연계, Threshold 기반 자동 판정에 강점이 있음
    * 향후 증가할 공공기관 클라우드 네이티브 전환 사업 대응을 위해 별도 육성이 필요함

#### nGrinder

* 포지션: `기존 자산 활용용 한시적 운영 도구`
* 선정 이유

    * 최근 수행 경험이 있고, Compose 기반 Controller-Agent 자산을 이미 보유함
    * 단기 프로젝트나 즉시 재현이 필요한 경우 빠르게 대응 가능함
* 제한 사항

    * 장기 표준 도구로 가져가기에는 유지보수성과 지속성 측면의 리스크가 존재함
    * 따라서 신규 장기 표준 후보보다는 현재 자산 활용 관점으로 한정하는 것이 적절함

### 3\. 고객 및 사업 유형별 적용 전략

#### 공공기관 기본 홈페이지 및 WEB-WAS-DB형 IaaS 사업

* 우선 도구: `JMeter`
* 적용 이유

    * 고객 설명력, 수행 표준화, 범용성 측면에서 가장 안정적임
    * 기존 방식과의 연속성 확보가 가능함

#### Kubernetes/MSA/API 중심 클라우드 네이티브 사업

* 우선 도구: `k6`
* 적용 이유

    * API 부하, 이벤트성 트래픽, 시나리오 제어, 자동화 측면에서 유리함
    * 향후 운영 자동화 및 품질게이트 체계와도 연결 가능함

#### 단기 즉시 대응 또는 기존 운영 자산 활용 프로젝트

* 우선 도구: `nGrinder`
* 적용 이유

    * 현재 자산을 활용해 빠르게 환경 재현 가능
    * 단기 수행 효율 확보 가능
* 단, 장기 표준 축과는 분리해서 관리해야 함

### 4\. 표준화 운영 방식

도구 표준화는 단순히 도구 하나를 정하는 것이 아니라, 문서, 템플릿, 가이드, 실행 자산까지 함께 표준화해야 한다. 현재 저장소는 `문서 표준`, `템플릿 표준`, `실행 자산 표준`, `운영 하네스`를 함께 관리하는 방식으로 정리했다. 이후 프로젝트별 대상 시스템 검증 결과는 각 도구의 `reports/<project>/<yyyymmdd>/` 경로로 축적한다.

### 5\. 최종 정리

현재 상황과 고객 특성, 기존 수행 경험, 향후 사업 방향을 종합하면 다음과 같이 정리할 수 있다.

* `JMeter`는 공공기관 대상 사업 수행과 조직 공통 표준 운영에 가장 적합한 도구인 것으로 생각된다.
* `k6`는 클라우드 네이티브/API 중심 사업 대응을 위한 전략 특화 도구로 육성할 필요가 있을 것으로 보인다.
* `nGrinder`는 현재 보유 자산과 수행 경험을 활용하는 한시적 운영 도구로 관리하는 것이 적절할 것으로 보인다.

따라서 표준화 방향은 `JMeter 중심 표준 운영 + k6 전략 특화 병행 + nGrinder 자산 제한 활용`으로 확정한다. 현재 저장소는 이 방향에 맞춰 문서, 템플릿, 실행 자산, 운영 하네스를 정리했으며, 실제 대상 시스템 부하 실행은 프로젝트별 승인 범위 안에서 수행한다.
