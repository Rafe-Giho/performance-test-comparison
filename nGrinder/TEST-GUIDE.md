# nGrinder 테스트 수행 가이드

## 기본 원칙

- nGrinder는 `즉시성`이 필요한 프로젝트에 쓴다.
- 장기 표준 증빙은 JMeter 또는 k6 기준으로 가져가고, nGrinder는 단기 실행 자산으로 본다.
- 새 프로젝트는 Compose와 스크립트 템플릿을 복제해서 시작한다.

## 실행 전에 알아야 하는 기초 지식

- `Controller`가 느리면 Agent가 충분해도 전체 테스트 운영이 매끄럽지 않다.
- `Agent` 수보다 중요한 것은 Agent 한 대가 실제로 몇 개의 thread를 안정적으로 처리할 수 있는지다.
- `vuser` 숫자가 커 보여도, 실제 대상 서버 병목과 부하발생기 병목을 구분해서 봐야 한다.

## 실무에서 중요한 원리

- nGrinder는 중앙 UI가 강점이므로, 여러 사람이 같은 절차로 테스트를 반복하기 좋다.
- 반대로 장시간 표준화 관점에서는 버전 재현성과 유지보수 리스크를 같이 관리해야 한다.
- 단일 고사양 호스트 통합 운영은 빠르지만 Controller와 Agent의 자원 간섭을 만든다.

## 적합한 상황

- 빠르게 테스트 환경을 띄워야 하는 경우
- 웹 UI 기반 운영이 필요한 경우
- 기존 조직 경험을 바로 활용해야 하는 경우
- 단기 프로젝트에서 일관된 운영 절차를 빠르게 강제해야 하는 경우

## 기본 수행 절차

1. Compose 템플릿 배포
2. Controller 접속
3. 프로젝트 생성
4. 스크립트 템플릿 선택 또는 복제
5. 데이터 파일 업로드
6. Agent 수 확인
7. Smoke 실행
8. Load/Stress/Spike/Soak 실행
9. 결과 리포트 저장

## 표준 시나리오

- 메인 접속
- 로그인
- 조회
- 상세
- 핵심 액션

## 표준 부하 패턴

- Smoke
- Load
- Stress
- Spike
- Soak

## 결과 저장 규칙

- 스크립트: `scripts/`
- 데이터: `data/`
- 결과: `reports/<project>/<yyyymmdd>/`

## 결과 해석 포인트

- TPS
- 응답시간
- 에러율
- Agent 자원 사용량
- 서버 자원 사용량

## 사용 제한

- 신규 장기 표준 증빙 문서는 nGrinder 중심으로 작성하지 않는다.
- archive 상태를 인지하고, 내부 단기 운영 목적에 한해 사용한다.
- 공통화 가능한 스크립트만 템플릿으로 남기고, 고객 전용 스크립트는 분리한다.
