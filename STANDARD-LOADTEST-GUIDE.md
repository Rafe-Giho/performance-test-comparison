# 표준 부하테스트 구축 및 수행 가이드

## 문서 등급

문서 등급: 팀 배포용 표준 운영 가이드

## 운영 환경

운영 환경: 클라우드 기반 테스트 환경

이 문서의 구축, 실행, 검증 기준은 클라우드 기반 테스트 환경을 전제로 한다. 테스트 인프라는 프로젝트별 클라우드 VM, 컨테이너, Kubernetes/NKS 자원으로 구성하고 운영 서비스와 논리적으로 분리한다.

이 문서는 도구 선택, 구축, 테스트 수행, 산출물 기준의 상위 표준이다. 팀 배포 전 [DOCUMENT-STANDARD.md](./DOCUMENT-STANDARD.md)의 기준과 운영 하네스 검증을 통과해야 한다.

## 목적

이 문서는 우리 조직이 `JMeter`, `nGrinder`, `k6`를 어떤 기준으로 선택하고, 어떤 방식으로 구축하고, 어떤 방식으로 테스트 증빙을 만들지 정리한 표준 가이드다.

이 문서의 목적은 두 가지다.

- 과장님 보고서의 결론을 실제 구축과 테스트로 증명할 수 있게 한다.
- 프로젝트마다 인력과 대상이 바뀌어도 같은 품질의 테스트를 반복할 수 있게 한다.
- 프로젝트별 일회성 파일이 아니라 범용 가이드와 템플릿을 먼저 만들고 재사용할 수 있게 한다.

## 최종 표준 정책

### 공통 표준

- `JMeter`

### 전략 특화

- `k6`

### 한시적 운영 자산

- `nGrinder`

## 표준 운영 원칙

- 공통 표준 도구는 `JMeter`를 기준으로 운영한다.
- 전략 특화 도구는 `k6`를 별도 축으로 육성한다.
- `nGrinder`는 현재 운영 자산으로만 활용한다.
- 모든 프로젝트는 범용 가이드와 템플릿을 먼저 적용하고, 프로젝트별 수정은 최소화한다.
- 테스트 절차, 스크립트 구조, 보고 형식은 템플릿화해서 재사용한다.
- 프로젝트 수행 중 새로 얻은 공통 패턴은 다시 표준 템플릿에 환류한다.

## 현재 검증 수준

| 도구 | 현재 완료 범위 | 프로젝트 투입 시 필수 확인 |
| --- | --- | --- |
| JMeter | 클라우드 VM/분산 운영 문서, 표준 `.jmx`, 실행 스크립트, Docker runner 자산, 하네스 검증 | 실제 VM 설치, 단일/분산 smoke, 대상 부하 실행 |
| k6 | NKS k6 Operator `4.3.2` 설치, CRD 확인, no-op `TestRun`, server dry-run, 적용 스크립트 dry-run | 실제 `BASE_URL`, Secret, payload, API/browser smoke, 본시험 |
| nGrinder | Docker Desktop Compose 기동, Controller HTTP `200`, Agent 연결 로그 | 클라우드 VM 배포, Controller UI smoke, 대상 부하 실행 |

이 표는 도구 우선순위가 아니라 현재 저장소 자산의 검증 범위다. 실제 대상 시스템 부하 실행은 승인된 환경값과 모니터링 연결 후 수행한다.

## 도구 선택 기준

| 조건 | 우선 도구 | 이유 |
| --- | --- | --- |
| 공공기관 대상 기본 웹서비스, WEB-WAS-DB형 IaaS | JMeter | 범용성, 익숙함, 보고서 설득력, 수행자 교체 대응력이 좋음 |
| Kubernetes, MSA, API Gateway, Ingress, CI/CD | k6 | 클라우드 네이티브 적합성, scenario/threshold, 자동화 친화성 |
| 즉시 가동이 최우선이고 내부 Compose 자산 활용 가능 | nGrinder | Controller-Agent를 빠르게 띄울 수 있음 |
| JDBC, JMS, FTP, TCP 등 혼합 프로토콜 | JMeter | 프로토콜 범위가 가장 넓음 |
| 이벤트성 API 폭주, arrival-rate, spike 모델링 | k6 | HTTP 기반 시나리오 표현력이 가장 좋음 |

## 표준 증빙 전략

도구를 비교하거나 보고서를 증명할 때는 아래 원칙을 따른다.

### 공통 비교 조건

- 동일한 테스트 대상
- 동일한 핵심 시나리오
- 동일한 부하 단계
- 동일한 성공 기준
- 동일한 메트릭 수집 범위

### 공통 시나리오 예시

1. 메인 페이지 또는 헬스체크 진입
2. 로그인
3. 목록 조회
4. 상세 조회
5. 핵심 이벤트 API 호출
6. 로그아웃 또는 종료

### 공통 템플릿 적용 원칙

- 모든 도구는 `health -> login -> list -> detail -> event` 흐름을 기본 템플릿으로 관리한다.
- 고객별 차이는 URL, 인증값, 데이터셋, 부하 수치만 바꾸는 방식으로 분리한다.
- 템플릿은 고객명이나 고객 고유 ID를 포함하지 않는다.
- 템플릿 변경은 개별 프로젝트 수정으로 끝내지 말고 재사용 가능한 변경인지 검토한다.

### 공통 부하 단계 예시

1. `Smoke`: 1~5분, 소수 사용자
2. `Load`: 목표 평균 트래픽 수준
3. `Stress`: 목표치를 초과한 상태
4. `Spike`: 짧은 시간 급증
5. `Soak`: 장시간 지속

### 공통 수집 메트릭

- TPS/RPS
- 평균 응답시간
- P90/P95/P99
- 에러율
- APDEX 또는 Threshold 판정
- CPU
- 메모리
- 네트워크
- DB connection, slow query, lock wait
- Ingress/Gateway/WAF 병목 여부

## 공통 인프라 원칙

- 테스트 인프라는 프로젝트마다 별도로 만든다.
- 운영과 물리적 또는 논리적으로 분리된 환경을 원칙으로 한다.
- 클라우드 네이티브 대상은 가능하면 대상 클러스터 밖 또는 전용 node group에서 실행한다.
- 결과 왜곡 방지를 위해 테스트 러너와 대상 서비스가 같은 리소스 풀을 강하게 공유하지 않도록 한다.

## 규모별 준비 기준

실무에서는 테스트 대상을 바로 대규모로 때리지 않는다. 보통 `소형 -> 중형 -> 대형 -> 이벤트성 급증` 순으로 단계적으로 올린다.

| 규모 | 대표 대상 예시 | 일반적인 준비 수준 | 공통 체크 포인트 |
| --- | --- | --- | --- |
| 소형 | 부서 홈페이지, 단순 조회 위주 웹사이트 | 단일 시나리오, 소수 사용자, 단일 또는 2대 수준의 러너 | 연결성, 기능 정상 여부, 에러 응답, 기본 리소스 사용량 |
| 중형 | 일반 공공기관 홈페이지, 게시판/검색/로그인 포함 | 핵심 시나리오 2~4개, 분산 실행 시작, DB/웹/WAS 모니터링 포함 | 병목 지점, 캐시 효과, 인증 처리량, 평균 부하 안정성 |
| 대형 | 통합 포털, 민원/첨부/내부 API 연계 포함 | 다중 시나리오, 다중 러너, 장시간 검증, 관측 스택 연동 | LB/Ingress/WAF 병목, DB connection, thread pool, 오토스케일 반응 |
| 이벤트성 급증 | 공지 공개, 접수 오픈, 신청/예약/결제 시작 시점 | spike 시나리오, arrival-rate 모델, 짧은 급증 + 회복 검증 | 순간 폭주, 에러율 급상승, 큐잉, 타임아웃, 복구 시간 |

### 도구별 규모별 시작점

| 규모 | JMeter | nGrinder | k6 |
| --- | --- | --- | --- |
| 소형 | 제어 1 + 엔진 1~2 | Controller 1 + Agent 1~2 | API 러너 1 |
| 중형 | 제어 1 + 엔진 2~4 | Controller 1 + Agent 2~4 | API 러너 2 또는 1대 고사양 |
| 대형 | 제어 1 + 엔진 4~8 | Controller 1 + Agent 3~5 또는 단일 고사양 + 보조 Agent | API 러너 2~4 또는 Kubernetes 분산 |
| 이벤트성 급증 | 다중 엔진 + 짧은 spike 패턴 | 다중 Agent + 중앙 제어 | arrival-rate 기반 러너 2~4, 필요 시 NKS |

### 규모별로 추가로 준비해야 할 것

#### 소형

- 대상 URL, 인증 방식, 핵심 경로만 정리
- 기본 로그와 CPU/메모리 모니터링만 있어도 시작 가능

#### 중형

- 시나리오별 비율 정의
- DB, WAS, LB 수준 모니터링 준비
- 테스트 데이터와 계정 분리

#### 대형

- 다중 러너 또는 다중 Agent 구조 설계
- 로그 수집, 대시보드, APM, DB 병목 확인 체계 준비
- 결과 저장 경로와 장시간 리포트 용량 계획

#### 이벤트성 급증

- 이벤트 시작 시점의 실제 사용자 행동을 모델링
- 캐시 warm-up과 본 spike 구간 분리
- 타임아웃, 재시도, 큐 적체, 오토스케일 반응시간을 같이 관측

## 클라우드 최소 시작 사양

아래 표의 사양은 `전체 합산 사양`이 아니라 `역할별 최소 시작 사양`이다.

- `Controller 1대`, `Agent 1대`, `엔진 1대`, `러너 1대`처럼 각 역할 인스턴스 또는 컨테이너 1개 기준으로 본다.
- 예를 들어 `nGrinder Agent 1대 = 4 vCPU / 8 GB`는 `Agent 한 개당` 기준이다.
- 따라서 `Controller 1대 + Agent 2대` 구성이라면 최소 시작 총량은 대략 `Controller 2/4 + Agent 4/8 x 2`로 계산한다.
- 다만 실제로는 `여러 역할을 한 인스턴스에 같이 올릴 수 있다.` 이 경우 표는 합산 기준이 아니라 `역할별 요구량을 한 호스트에 적재할 수 있는지`를 보는 참고값이다.
- 즉, 단일 고사양 인스턴스 1대에 `nGrinder Controller + Agent`를 같이 올려도 무방하지만, 그 경우에도 Controller와 Agent의 요구량을 합산해서 여유 자원까지 계산해야 한다.

| 도구 | 역할 | 최소 시작 사양 | 권장 시작 사양 |
| --- | --- | --- | --- |
| JMeter | 제어/작성 노드 | 2 vCPU / 4 GB RAM / 30 GB SSD | 4 vCPU / 8 GB RAM / 50 GB SSD |
| JMeter | 엔진 1대 | 4 vCPU / 8 GB RAM / 50 GB SSD | 8 vCPU / 16 GB RAM / 100 GB SSD |
| nGrinder | Controller | 2 vCPU / 4 GB RAM / 50 GB SSD | 4 vCPU / 8 GB RAM / 100 GB SSD |
| nGrinder | Agent 1대 | 4 vCPU / 8 GB RAM / 50 GB SSD | 8 vCPU / 16 GB RAM / 100 GB SSD |
| k6 | API 러너 1대 | 2 vCPU / 4 GB RAM / 30 GB SSD | 4 vCPU / 8 GB RAM / 50 GB SSD |
| k6 | Browser 러너 1대 | 4 vCPU / 8 GB RAM / 30 GB SSD | 8 vCPU / 16 GB RAM / 50 GB SSD |

### 예시 해석

- `JMeter 제어 1 + 엔진 2` 최소 시작 총량
  - 제어 노드 1대: `2 vCPU / 4 GB`
  - 엔진 2대: 각 `4 vCPU / 8 GB`
  - 분리 인스턴스 기준 총량: `10 vCPU / 20 GB RAM`
- `nGrinder Controller 1 + Agent 1` 단일 인스턴스 통합 운영 예시
  - 표 기준 합산 최소치는 `6 vCPU / 12 GB RAM`
  - 실제 운영은 JVM heap, OS 여유 메모리, 로그 저장 공간을 더 봐야 하므로 `8 vCPU / 16 GB` 이상부터 시작하는 편이 안전하다.
- `k6 API 러너 2대`
  - 러너 1대당 `2 vCPU / 4 GB`
  - 총량은 `4 vCPU / 8 GB`가 아니라 `2 vCPU / 4 GB` 인스턴스 2대 또는 그 이상으로 보는 것이 맞다.

### nGrinder 단일 고사양 인스턴스 예시 해석

아래와 같은 `Controller + Agent` 통합 Docker Compose 구성은 표준 최소 사양과 모순되지 않는다.

- 해석 기준
  - 표의 `Controller 1대`, `Agent 1대`는 각각의 역할 요구량이다.
  - 단일 고사양 인스턴스에 두 역할을 함께 올리면 `역할별 요구량의 합 + OS 여유 자원 + Docker 여유 자원`으로 계산한다.
- 예시 해석
  - `32 vCore / 32 GB RAM` 단일 인스턴스
  - Controller 1개 + Agent 1개
  - Agent JVM heap `20 GB`, 컨테이너 limit `24 GB`
  - 이 구성은 `최소 시작 사양`이 아니라 `고사양 통합 운영 구성`에 해당한다.
- 판단
  - 웹/API 중심 테스트에서 문제 없이 돌았다면 이상한 것이 아니다.
  - 오히려 최소 시작 사양보다 훨씬 큰 자원을 단일 호스트에 집중한 사례로 이해하면 된다.
- 주의
  - 이 방식은 빠르고 단순하지만, Controller와 Agent가 같은 호스트를 공유하므로 장애 분리와 자원 간섭 측면에서는 분리 구성보다 불리하다.

## 표준 버전 관리 원칙
